# GitHub Workflow Setup

This document is the operational source of truth for Ambit's pull request, dependency, release, and updater workflows.

## Toolchain Policy

- Node.js: `24`, declared in `.node-version` and used by every workflow.
- pnpm: `11.5.3`, pinned by the `packageManager` field in `package.json`.
- Rust build toolchain: `1.96.0`, pinned in `rust-toolchain.toml` and used for audits and releases.
- Rust minimum supported version: `1.88.0`, declared in `src-tauri/Cargo.toml` and exercised by the Windows PR job.
- GitHub Actions: every external action is pinned to a full commit SHA. The release tag remains in a comment for Dependabot and human review.

Rust `1.77.2` is no longer a valid minimum. The current lockfile contains Edition 2024 crates and dependencies that require Rust `1.88.0`.

## Pull Request Checks

Protect `main` with these required status checks:

- `frontend-checks`
- `rust-tests`
- `dependency-audits`

`frontend-checks` validates the Conventional Commit PR title, version consistency, lint, TypeScript, Vitest coverage, and the production bundle guard.

`rust-tests` runs on Windows with Rust `1.88.0`, restores the Rust cache, verifies generated bindings, runs Rust tests, and performs the Tauri no-bundle build.

`dependency-audits` is always present as a stable check:

- npm manifest or lockfile changes run `pnpm audit --prod` without installing frontend dependencies.
- Cargo manifest, lockfile, or Rust audit-wrapper changes run `cargo audit`.
- Dependabot or workflow changes run both audits.
- Scheduled and manually dispatched runs execute both audits.
- Unrelated PRs report that audits were skipped and pass the check.

Do not require an always-green dependency-review placeholder. Enable the real `actions/dependency-review-action` only after the repository is public or GitHub Advanced Security makes it available.

## Repository Rules

GitHub exposes branch protection and repository rulesets for private repositories only on a paid plan. If Ambit remains private on GitHub Free, upgrade the account before treating required checks as enforced policy.

The repository merge settings should allow squash merges only and automatically delete merged branches.

When branch protection is available, configure `main` for a strict solo-maintainer workflow:

- Require a pull request before merging, with zero required approvals.
- Require branches to be up to date before merging.
- Require the three status checks listed above.
- Require conversation resolution and linear history.
- Apply the rules to administrators and disable bypass.
- Block force pushes and branch deletion.

PR titles must follow Conventional Commits, for example `feat: add viewer zoom`, `fix(ui): align toolbar`, or `feat(api)!: remove legacy mode`.

## Actions Policy And Billing

Keep default `GITHUB_TOKEN` permissions read-only except for the narrowly scoped release job permissions already declared in workflow YAML.

Under **Settings > Actions > General**, use the selected-actions policy, require full-length commit SHAs, and allow these action repositories:

- `actions/checkout`
- `actions/setup-node`
- `actions/upload-artifact`
- `actions/create-github-app-token`
- `pnpm/action-setup`
- `dtolnay/rust-toolchain`
- `swatinem/rust-cache`
- `googleapis/release-please-action`
- `tauri-apps/tauri-action`

Ambit is private and uses Windows runners, so confirm an Actions spending limit and valid payment method under the account's billing and metered-usage settings. A private-repository quota or payment block can reject workflows before any job starts.

## Release Please GitHub App

Release Please must use a short-lived GitHub App installation token. There is no PAT or `GITHUB_TOKEN` fallback.

1. Create a GitHub App named for Ambit release automation.
2. Disable webhooks unless they are needed for another purpose.
3. Grant repository permissions:
   - Contents: read and write
   - Pull requests: read and write
   - Issues: read and write
4. Install the App only on `AsuraAce/ambit`.
5. Add the App client ID as the repository variable `RELEASE_APP_CLIENT_ID`.
6. Generate a private key and add its full PEM value as the repository secret `RELEASE_APP_PRIVATE_KEY`.
7. Delete the old `RELEASE_PLEASE_TOKEN` secret after the App-backed workflow succeeds.

The workflow further restricts each generated token to Contents, Pull Requests, and Issues write access. App-created release PRs and tags can trigger downstream workflows.

Release Please remains the source of truth for version bumps, `CHANGELOG.md`, release PRs, tags, and GitHub Releases.

## Updater Secrets

- Create a GitHub Environment named `updater-signing` without required reviewers.
- Store `TAURI_SIGNING_PRIVATE_KEY` only as an `updater-signing` environment secret. Use the complete CI key string generated by Tauri.
- Store `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` only as an `updater-signing` environment secret when the signing key has a non-empty passphrase.
- Delete repository-level `TAURI_SIGNING_PRIVATE_KEY` and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` after the environment-backed workflows are merged and the environment secrets are installed.

The matching public key is embedded in `src-tauri/tauri.conf.json`. Keep the private key outside the repository and outside synced public folders.

Do not create release tags while updater signing secrets still exist at repository scope. Environment-scoped signing secrets prevent historical workflow revisions from reading the key when an old commit is tagged.

Use **Actions > updater-signing-preflight > Run workflow** before publishing. The successful log must end with `Updater signing preflight passed.`

## Release Flow

1. Work on a feature branch and open a PR into `main`.
2. Wait for all three required checks.
3. Squash merge so the PR's Conventional Commit title becomes the commit on `main`.
4. Confirm `release-please` opens or refreshes its release PR and that the PR receives the required checks automatically.
5. Review the generated version and changelog.
6. Merge the release PR only when a release should be published.
7. Confirm the generated `vX.Y.Z` tag starts `release.yml`.

The release workflow currently publishes Windows NSIS and MSI artifacts, updater signatures, and `latest.json`. It copies the matching version section from `CHANGELOG.md` into the updater metadata so installed apps can show the same release notes. Release tags must use the exact `vX.Y.Z` format, match `package.json`, `src-tauri/tauri.conf.json`, `src-tauri/tauri.dev.json`, and `src-tauri/Cargo.toml`, and point to a commit reachable from `main`.

Release Please remains the normal and preferred tag creator. Do not create manual release tags except for recovery work after confirming the tag target is already on `main` and all version files match.

## Experimental Unix Builds

Use **Actions > experimental-unix-builds > Run workflow** when Linux or macOS community testers need fresh artifacts. This workflow is manual only and uploads short-lived workflow artifacts for packaging validation.

- It does not publish GitHub Releases, release assets, updater signatures, or `latest.json`.
- It disables updater artifact generation and does not require `TAURI_SIGNING_PRIVATE_KEY`.
- Linux currently builds AppImage and Debian packages on `ubuntu-22.04`.
- macOS currently builds an unsigned, non-notarized DMG on `macos-latest`.

Keep tester instructions in `docs/experimental-unix-builds.md`. Do not treat successful experimental packaging as official platform support until install, launch, data path, keyring, file-open, thumbnail, and updater behavior have explicit platform evidence.

## Startup-Failure Recovery

If a workflow reports `startup_failure` with zero jobs:

1. Authenticate GitHub CLI with `gh auth login -h github.com` and confirm `gh auth status`.
2. Inspect the run annotation with `gh run view <run-id>` before editing YAML.
3. Check Actions billing and metered usage for a quota or payment block.
4. Check the repository Actions policy and selected-action allowlist.
5. Validate workflow YAML locally with `actionlint`.
6. Rerun the failed jobs only after the GitHub-side rejection is resolved.

Historical runs `27337343607` (`pr-ci`) and `27337343030` (`dependency-security`) both failed before creating jobs. That evidence points to GitHub policy, billing, or repository state rather than a failing test command.

## Local Git Hygiene

- Set repository-local `fetch.prune=true`, `fetch.pruneTags=true`, and `pull.ff=only`.
- Review gone-tracking branches individually before deletion; a missing upstream does not prove that unique local commits are disposable.
- Review all linked worktrees before deleting a local branch.

After CI and the GitHub App are working, inspect the existing Release Please PR. Reuse it only if Release Please can refresh it cleanly; otherwise close the obsolete PR, delete its branch, and dispatch a fresh release PR. Do not merge or publish it merely to test automation.

## Updater Risks

- Losing the private updater key prevents existing installed builds from accepting future updates without a migration strategy.
- A public/private key mismatch causes downloaded updates to be rejected.
- While the repository and release assets are private, installed apps cannot consume GitHub's unauthenticated public release feed.
